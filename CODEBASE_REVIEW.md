# PRS 코드베이스 종합 분석 및 평가

## 1) 한 줄 요약
- **기능 완성도는 높은 편**이며(크롤링 + 다수 점검 + 리포트/인터랙티브 UX), 실사용을 고려한 안전장치가 포함된 구조입니다.
- 반면 **테스트 자동화 부재**, 일부 **설계 결합도**, 그리고 **성능/정확도 튜닝 포인트**가 남아 있어, 다음 릴리스에서 품질 안정성 강화를 권장합니다.

## 2) 아키텍처 관점 평가

### 장점
- CLI 진입점(`main -> cmd`)과 실행 흐름(`scan`, `interactive`)이 비교적 명확합니다.
- 점검 항목이 `registry.DefaultChecks()`에 중앙 등록되어 확장성이 좋습니다.
- `checks/context`로 공통 스캔 컨텍스트를 전달해 체크 구현체의 일관성을 유지합니다.
- 결과 모델(`report.Finding`)이 단일 타입으로 정리되어 출력(JSON/HTML/콘솔) 파이프라인과 잘 맞습니다.

### 아쉬운 점
- `internal/app/scan/scan.go`에 많은 책임이 집중(타깃 검증, 크롤링 orchestration, 동시성 제어, 집계/중복제거, 출력/저장 호출)되어 있습니다.
- 일부 프롬프트 흐름(예: HTML 리포트 저장 여부)과 실행 로직이 결합되어 비대화형 실행 자동화에 제약을 줄 수 있습니다.

### 개선 제안
- `scan.go`를 다음 단위로 분리:
  1. `target validation/normalization`
  2. `discovery(crawl)`
  3. `executor(concurrency + retry/timeout)`
  4. `aggregation/reporting`
- `--non-interactive` 혹은 `--html` 명시 플래그 중심으로 자동화 친화 모드 강화.

## 3) 동시성/성능 관점 평가

### 장점
- 스캔 실행부와 체크 실행부 모두 semaphore 기반 동시성 상한을 둬 폭주를 방지합니다.
- HTTP 클라이언트를 공유해 커넥션 풀링 이점을 활용합니다.
- 크롤러는 링크 수/바디 크기/최대 타깃 수 제한이 있어 현실적인 보호 장치가 있습니다.

### 리스크
- 체크 단위 에러가 조용히 누락되는 구간이 있어(에러 집계 미흡) 문제 원인 추적성이 떨어질 수 있습니다.
- 고정 동시성(예: 5)은 환경/타깃 특성에 따라 비효율적일 수 있습니다.

### 개선 제안
- 체크 실패 원인을 `results + errors` 구조로 수집해 최종 summary에 노출.
- `--concurrency` 플래그 제공 및 기본값 자동 튜닝(코어 수/RTT 기반) 검토.

## 4) 보안 스캐너 품질 관점 평가

### 장점
- 패시브/액티브 모드 구분이 명확하고, 액티브 스캔 전 사용자 확인 절차가 있어 안전합니다.
- CORS, 헤더, 세션, API, 인젝션, SSRF, 컴포넌트 등 범위가 넓습니다.
- 탐지 결과에 `severity`, `confidence`, `fix`, `evidence`가 있어 활용성이 높습니다.

### 주의할 점
- 일부 항목은 휴리스틱 기반이라 오탐/미탐 가능성이 큽니다(특히 인젝션/애플리케이션 로직 계열).
- 점검 신뢰도를 뒷받침할 테스트 fixture/회귀 검증 체계가 현재 보이지 않습니다.

### 개선 제안
- 취약점별 로컬 테스트 타깃(의도적 취약 앱) 기반 회귀 테스트 세트 구축.
- 결과 스키마에 `detection_method`, `request_sample`, `response_sample_hash` 같은 포렌식 친화 필드 확장 고려.

## 5) 유지보수성/품질 관점 평가

### 현재 상태
- 모듈 분리는 전반적으로 잘 되어 있으나, 대형 파일(메시지/스캔 오케스트레이션) 중심으로 가독성 비용이 큽니다.
- 테스트 코드가 없어 리팩터링 리스크가 높습니다.

### 우선순위 제안
1. **P0**: 핵심 로직 단위 테스트 추가(`normalize`, `dedupe`, severity policy, JSON export)
2. **P1**: 스캐너 통합 테스트(모의 서버 기반)
3. **P1**: 에러 리포팅/관측성 개선
4. **P2**: 설정/플래그 체계 정리(`interactive vs non-interactive`)

- 다음 단계의 핵심은 **테스트 체계 확립 + 책임 분리 리팩터링 + 관측성 강화**입니다.
